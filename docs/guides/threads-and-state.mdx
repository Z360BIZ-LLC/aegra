title: "Threads and state"
description: "Manage conversations, inspect state, and work with checkpoints."

Threads are the core persistence unit in Aegra. Each thread represents a conversation with its own state, message history, and checkpoints. Runs execute within threads, and state is automatically persisted after each node execution.

## Creating threads

```python
import asyncio
from langgraph_sdk import get_client


async def main():
    client = get_client(url="http://localhost:2026")

    # Create a thread
    thread = await client.threads.create()

    # Create with metadata
    thread = await client.threads.create(
        metadata={"user_name": "Alice", "session": "onboarding"},
    )

    # Create with a specific ID (idempotent)
    thread = await client.threads.create(
        thread_id="my-thread-123",
        if_exists="do_nothing",  # Don't error if it already exists
    )


asyncio.run(main())
```

## Thread status

Threads have a status that reflects their current state:

| Status | Meaning |
|--------|---------|
| `idle` | No active run, ready for new input |
| `busy` | A run is currently executing |
| `interrupted` | A run paused for human input |
| `error` | The last run ended with an error |

```python
thread = await client.threads.get(thread_id)
print(thread["status"])  # "idle", "busy", "interrupted", "error"
```

<Note>
  The snippets below assume you are inside an `async def` function with an initialized `client` — see the example above.
</Note>

## Getting thread state

The state contains the current values of all state fields, information about what nodes will execute next, and any pending interrupts.

```python
state = await client.threads.get_state(thread_id)

# Current state values (your State schema fields)
print(state["values"])

# Next nodes to execute (empty if completed)
print(state["next"])

# Pending tasks
print(state["tasks"])

# Active interrupts (if thread is interrupted)
print(state["interrupts"])

# Checkpoint info
print(state["checkpoint"]["checkpoint_id"])
```

## Updating thread state

You can modify thread state directly, for example to inject data or correct values:

```python
await client.threads.update_state(
    thread_id,
    values={"messages": [{"type": "human", "content": "Injected message"}]},
    as_node="agent",  # Apply update as if it came from this node
)
```

## Checkpoint history

Every state change creates a checkpoint. You can browse the full history:

```python
# Get recent checkpoints
history = await client.threads.get_history(thread_id)

for entry in history:
    print(f"Checkpoint: {entry['checkpoint']['checkpoint_id']}")
    print(f"  Values: {entry['values']}")
    print(f"  Next: {entry['next']}")
```

### Get state at a specific checkpoint

```python
# Go back in time to a specific checkpoint
state = await client.threads.get_state(
    thread_id,
    checkpoint={"checkpoint_id": "your-checkpoint-id"},
)
```

This is useful for debugging, replaying, or branching conversations from a previous point.

## Searching threads

Find threads by status or metadata:

```python
# Search by metadata
threads = await client.threads.search(
    metadata={"user_name": "Alice"},
)

# Search by status
threads = await client.threads.search(
    status="interrupted",
)

# With pagination
threads = await client.threads.search(
    limit=20,
    offset=0,
)

# Filter to specific thread IDs
threads = await client.threads.search(
    ids=["thread-1", "thread-2"],
)
```

### Projecting and extracting fields

To render a conversation list (title + last-message preview) in a **single** call —
without fetching each thread's full state — use `select` and `extract`:

```python
# `select` whitelists the returned fields (smaller responses)
threads = await client.threads.search(
    select=["thread_id", "status", "metadata", "values"],
)

# `extract` pulls specific JSON paths into an `extracted` field
threads = await client.threads.search(
    select=["thread_id"],
    extract={
        "title": "values.messages[0].content",
        "last_msg": "values.messages[-1].content",
    },
)
# -> [{"thread_id": "...", "extracted": {"title": "...", "last_msg": "..."}}, ...]

# `interrupts` returns the SDK shape {task_id: [Interrupt]} for the latest checkpoint
threads = await client.threads.search(
    select=["thread_id", "interrupts"],
)
# -> [{"thread_id": "...", "interrupts": {"<task_id>": [{"value": ..., "id": ...}]}}, ...]
```

- **`select`** — supported fields: `thread_id`, `status`, `created_at`, `updated_at`,
  `metadata`, `values`, `interrupts`. Omit `select` to get the full thread shape
  (backward compatible).
- **`extract`** — maps an alias to a JSON path (dot keys + array indexing incl. `[-1]`),
  capped at 10. Validation matches LangGraph: the path **root** must be `values`,
  `metadata`, `config`, or `interrupts` (so `interrupts[-1].value` is accepted, not just
  dotted forms). Anything that doesn't resolve — a malformed sub-path, a missing key, an
  out-of-range/typed index — yields `null` (never a 422). `config` is accepted for parity
  but always resolves `null` (Aegra doesn't materialize a config column).
  Note `interrupts` is a **mapping keyed by task id** (matching the SDK), so `interrupts[-1]`
  resolves to `null` — use `select:["interrupts"]` and read the map, or
  `interrupts.<task_id>[0].value`.
- `values`/`interrupts` are served from the thread's **materialized state** — the
  graph's logical state (`aget_state`), persisted to the thread row on run completion
  and state update. This matches `GET /threads/{id}/state` for every agent (including
  middleware/subgraph agents whose messages aren't recoverable from raw checkpoint
  blobs), with no per-thread graph replay and no N+1.
- Because the state is real JSONB, `extract` paths are resolved **in Postgres**
  (`jsonb_path_query_first`): the DB returns only the referenced value (e.g. just the
  last message content), not the whole state. The full state column is transferred only
  when you `select` the entire `values`/`interrupts`.

<Note>
Threads created before this feature have no materialized state. On startup the
server runs a **one-time backfill** automatically — guarded by an advisory lock
(single runner) and a completion marker (never re-runs), and rate-limited so it
won't stampede the DB. New threads materialize on run completion, so after the
first pass nothing else is needed. To disable it and run manually instead, set
`BACKFILL_THREAD_STATE_ON_STARTUP=false` and run `aegra db backfill-thread-state`
(supports `--batch-size` / `--sleep`).
</Note>

<Note>
The SDK fields `select=["config" | "context"]` and filtering by state `values=...` are
not yet supported and return `422`. Sorting by `state_updated_at` also returns `422`
(use `updated_at`).
</Note>

## Listing threads

```python
threads = await client.threads.search()
for thread in threads:
    print(f"{thread['thread_id']}: {thread['status']}")
```

## Deleting threads

Deleting a thread cancels any active runs and removes all state:

```python
await client.threads.delete(thread_id)
```

## Thread metadata

Threads automatically track metadata about their usage:

- `owner`: User identity at creation time (mirrors `user_id` on the Thread object, which is used for access control)
- `assistant_id`: Last assistant used
- `graph_id`: Last graph executed

You can add your own metadata at creation or update time:

```python
# Update metadata
await client.threads.update(
    thread_id,
    metadata={"priority": "high"},
)
```

## User isolation

When authentication is enabled, threads are automatically scoped to the authenticated user. Users can only see and interact with their own threads.
